## Abstract

Two profiles can guarantee type safety and resource safety for C++ application code without abandoning compatibility or the zero-overhead principle.

The concept of and rationale for Profiles<sup>[1, 2, 3]</sup> are summarized and some suggested alternatives are mentioned. The type-safety and resource-safety profiles are presented in some detail as an example to illustrate the specification and implementation requirements of two of the most essential and difficult profiles.

This is a description of Profiles in general terms aimed at understanding, not a standards text aimed at standards experts only. However, it is sufficiently detailed to be translated in "standardese" with relatively few technical changes. What is described has been prototyped<sup>[4, 5, 6]</sup> and some parts used in the form of guidelines.

R1 adds evidence of need from the regulatory landscape, the committee record, implementation experience, a teaching story, success criteria, and testable predictions. A plan for progress is outlined (Section 10).

## 1. Introduction

My initial motivation to develop C++ was to combine the best of C and Simula:

- C's ability to access hardware and manipulate data without overhead.
- Simula's ability to organize programs and guarantee type safety.

To this, I immediately added

- constructors and destructors to provide the ability to eliminate resource leaks and uninitialized objects.
- uniform treatment of built-in types and user-defined types to lay the foundation of generic programming.

C compatibility concerns and the zero overhead principle prevented me (and later the C++ standards committee) from enforcing complete type and resource safety. I (still) consider compatibility/stability and the zero-overhead principle essential for the evolution of C++ and its foundational libraries. Where this is not adhered to, problems emerge.

However, not everybody needs every old feature and not everybody needs ultimate performance. In fact, many (probably most) C++ users don't need all of the constraints and facilities needed to conform to either or both compatibility and performance ideals. Also, they often have other concerns, as do I for various projects. This is the motivation for all coding guidelines and for the (still evolving) Profiles design<sup>[3, 7, 6]</sup>. In particular, we can design a Profile to guarantee type safety and resource safety (the absence of resource leaks) without imposing inessential overheads.

These guarantees are the ones I want for most application code:

- **Type safety**: Every object is used exclusively according to its definition.
- **Resource safety**: No resource is leaked.

We can achieve that by careful use of contemporary C++, but I want a Profile to guarantee it, as do important C++ users. Otherwise, the approach won't scale. This implies that code that needs to violate that guarantee or can't be proven not to must be isolated. Examples of such necessary code include the implementation of key abstractions (e.g., `vector`). It also implies that the guarantee must be opt-in because essentially no existing code consistently meets the requirements needed to provide this guarantee (even correct code because "correct" doesn't mean "provably correct").

The need for these guarantees is not hypothetical. The WG21 Direction Group - Wong, Hinnant, Orr, Stroustrup, and Vandevoorde - unanimously endorsed profiles as the mechanism in [P2759R1](https://wg21.link/p2759r1)<sup>[27]</sup>, stating: "We now support the idea that the changes for safety need to be not just in tooling, but visible in the language/compiler and library." The author presented the profiles direction to SG23 and EWG at Issaquah in February 2023 ([P2816R0](https://wg21.link/p2816r0)<sup>[3]</sup>). The resulting vote was 47 for and 2 against. Since then, the committee has voted repeatedly in favor: SG23 at Wroclaw gave strong consensus (18-1-0) to the initialization profile<sup>[40]</sup>, and at Hagenberg in February 2025, EWG voted to pursue a language safety white paper covering Erroneous Behavior, Profiles, and Contracts<sup>[41]</sup>. Microsoft stated in [P3498R0](https://wg21.link/p3498r0)<sup>[38]</sup>: "We consider that the proposals along the lines of C++ Profiles are promising." Six senior committee members co-signed [P3970R0](https://wg21.link/p3970r0)<sup>[21]</sup> as a call to action.

The oft-cited statistic that 70% of critical vulnerabilities are memory-safety issues requires context. The Direction Group noted that "the US agencies lumping together C and C++ is likely conflating two languages' properties"<sup>[27]</sup>. As Sutter documented<sup>[35]</sup> and [P3500R1](https://wg21.link/p3500r1)<sup>[36]</sup> confirmed, "the consensus among security experts is that even if we could somehow make C++ a memory-safe language, that would not lead to 70% fewer CVEs, data breaches, and ransomware attacks." The 70% is of the subset of CVEs related to language safety; most of MITRE's 2023 "most dangerous weaknesses" were not memory-safety-related. Google's Android data shows that five-year-old code has 3.4x to 7.4x lower vulnerability density than new code<sup>[37]</sup> - the problem is not that C++ is inherently unfixable but that it is too easy by default to write code that violates type, bounds, initialization, and lifetime safety. These four categories are the genuine pain points, and they are precisely what profiles enforce. A type-safety profile does not need the 70% figure to be accurate - it addresses the actual problem. However, policymakers have acted on the 70% narrative regardless of its accuracy. The regulatory threat described in Section 1.2 is real. Correcting the statistic does not eliminate the need; it sharpens the response.

The fundamental assumptions behind the Profiles design are

- Some people need guaranteed properties of their code.
- Not everybody has the same needs.
- It is impossible to prove an arbitrary program safe for any reasonable definition of "safe" in a language able to manipulate hardware and implement fundamental abstractions.
- A guarantee must be validated by software, not just by a review by developers.
- A desired guarantee must be explicitly requested in code.

This implies that we need a framework that allows programmers to request a variety of guarantees and prevent the use of features deemed unsafe for the requested guarantees.

The fundamental technique for implementing a profile is to subset a superset of ISO standard C++<sup>[8, 9]</sup>:

- **Superset**: Extend the C++ to be used with the abstractions needed to conveniently express what needs to be expressed for a given application area.
- **Subset**: Restrict the set of facilities to be used in application code to what can be used while maintaining the desired guarantee (leaving "unsafe" uses to the implementation of the abstractions).

Supersetting is done by adding libraries and eliminating undefined behavior (supported by run-time checks where necessary). Subsetting is done through static analysis (ideally by the compiler).

Subsetting alone cannot deliver many of the most desired guarantees because the implementation of the abstractions requires use of techniques that cannot be verified (e.g., pointer arithmetic and linked structures).

Supersetting alone can't deliver many of the most desired guarantees because the low-level features would still be available to break them.

By supersetting, we create a language that's easier for analyzers (e.g., compilers) to understand. Use of banned features outside explicitly trusted sections of code is simply considered errors. This solves the problem that large chunks of C++ (especially old-style code) are too complex to analyze efficiently relative to a requested guarantee.

By specifying requirements as enforceable profiles, we avoid having to rely on developers following guidelines and sprinkling annotations all over the code. That approach is labor intensive, error prone, and basically not feasible at scale. It doesn't provide the guarantees required by many user organizations.

Profiles do not interfere with traditional C++ design and implementation techniques (e.g., OO, GP, and functional styles) except when we define a profile to do so and explicitly require that profile. Rather, a good profile eliminates misuses of those techniques.

The result of compiling a TU using profiles is ISO standard C++ that can be linked with other TUs<sup>[6]</sup>.

A profile cannot change the semantics of a program beyond defining the meaning of some forms of undefined behavior. For example, signed arithmetic overflow is UB so a profile can define it to be wraparound like unsigned arithmetic (though I wouldn't do that), to be saturated arithmetic, or to throw an exception. However, wraparound of unsigned arithmetic is well-defined so a profile cannot define it to be anything else. This implies that a program without UB will behave identically when compiled with or without a profile enforced. The aim of key profiles is to insert run-time checks to ensure that an error action is triggered rather than reaching the point of UB.

If a profile needs a language change (e.g., the addition of a new standard attribute), then that change must be done to the language rather than just to the profile.

Profiles are not just about "safety" (for a variety of kinds of safety). They address the widespread need for support of a better style of code (for a variety of definitions of "better") from both developers and educators.

Some, but not all, profiles will be part of the standard. Profiles is an open framework so that profiles serving specific needs can be added "locally." One limiting factor here will be the lack of a standard internal representation of C++ (e.g., IPR<sup>[10, 11]</sup>, Clang AST<sup>[12]</sup>).

### 1.1 Alternatives to Profiles

Why do we need a framework? After all, people can and do address problems with libraries, static analysis, compiler options, pragmas, etc. That variety is the problem. These features tend to address limited problems, be platform specific, be incompatible, require detailed understanding of compilation and build systems, be poorly specified, be non-portable, require consistent use of coding guidelines, be non-standard, and the like. The variety of genuinely helpful facilities that differ in their style of invocation is a recipe for chaos. For the key facilities that we want to be part of the standard, we need a standard way to specify them and document their use in the code. For all facilities that are not simply sets of libraries, we need a standard way of requesting their use.

Offering long lists of specific tests and restrictions for developers to select from are ever-popular among people who desire maximal freedom of choice. However, having hundreds of thousands of developers choosing from a set of (say) 70 options leads to unreadable code, incompatible code, and sets of options that do not add up to a guarantee that can be understood, named, and remembered. Languages, libraries, and systems have failed, being rejected in favor of simpler alternatives. Part of the aim of profiles is to reduce the set of alternatives to something manageable; to move management of complexity from implementation details to language and library design.

Several competing approaches address the same problem space. Each deserves consideration at its strongest.

A **borrow-checker or ownership model** (as in Rust, or proposed for C++ by cppfront and Circle) provides stronger guarantees with compile-time enforcement and no escape hatch in safe code. The tradeoff is fundamental incompatibility with C++'s object model and lifetime model. Profiles preserve full backward link compatibility and do not require a new type system - existing C++ code can adopt profiles incrementally without rewriting data structures or abandoning established libraries.

**Contracts** ([P2900R14](https://wg21.link/p2900r14)<sup>[43]</sup>) express preconditions and postconditions with different ergonomics. Contracts are complementary to profiles: a contract assertion checks a specific predicate at a specific call site, while a profile enforces structural guarantees across an entire translation unit. The Hagenberg white paper explicitly covers both<sup>[41]</sup>.

**Standardizing analysis rules without a framework** avoids the language-change cost. But without the framework, there is no composability, no `[[enforce]]`, no portability guarantee, and no way to name a guarantee. Each implementation would enforce its own interpretation, and code that passes one analyzer would fail another. The framework is what makes the guarantee portable.

**Doing nothing** and relying on steadily improving tooling is no longer viable. Improvement is fragmented, non-portable, and does not add up to a named guarantee. More critically, it does not satisfy the regulatory demand for a demonstrable memory-safety roadmap (Section 1.2). Government agencies have set deadlines and the underlying policy documents remain in force regardless of which administration enforces them.

By defining sets of optional libraries, tests, and implementation alternatives, rather than defining guarantees as language features, we could not just avoid making potentially erroneous design decisions, we could also avoid problems of evolving guarantees. If we in the future find a profile needs changing (as we surely will), we have exactly the same options as when we find that a language feature or a standard library needs a change:

- Vote in a change, trying to minimize impact on existing code (e.g., making it a pure extension).
- Make a new profile with a new name and potentially deprecate the old one (as we did when replacing `auto_ptr` with `unique_ptr`).

Evolution is difficult, but none of the problems are unique to profiles. Making design decisions is a key part of language design.

### 1.2 The Regulatory Landscape

Beginning in November 2022, a series of government actions created an existential threat to C++:

- **November 2022**: The NSA published a "Software Memory Safety" cybersecurity information sheet<sup>[29]</sup> recommending that organizations shift to memory-safe languages, citing Microsoft and Google data that approximately 70% of exploitable vulnerabilities stem from memory-safety issues.
- **December 2023**: CISA, NSA, FBI, and Five Eyes allies published "The Case for Memory Safe Roadmaps"<sup>[30]</sup>, urging software manufacturers to create transition plans away from C and C++.
- **February 2024**: The White House ONCD published "Back to the Building Blocks: A Path Toward Secure and Measurable Software"<sup>[31]</sup>, explicitly naming C and C++ as memory-unsafe and recommending migration to Rust, Go, Java, C#, and Swift.
- **October 2024**: CISA's "Product Security Bad Practices" guidance<sup>[32]</sup> set a January 1, 2026 deadline for manufacturers to publish memory-safety roadmaps.
- **January 2026**: OMB rescinded M-22-18<sup>[34]</sup> (the mandatory secure software attestation form), replacing it with agency-specific risk assessments.

The rescission of M-22-18 removes the current enforcement mechanism, but it does not withdraw the underlying policy direction. Executive Order 14028<sup>[33]</sup> remains in effect. The CISA guidance, the NSA information sheet, the ONCD report, and the Five Eyes joint guidance all remain published. A future administration can reinstate or strengthen the mandate overnight.

The author first responded to the NSA guidance in [P2739R0](https://wg21.link/p2739r0)<sup>[28]</sup> (2022): "A call to action: Think seriously about 'safety'; then do something sensible about it." In a February 2025 message to the SG23 mailing list, citing CISA's Product Security Bad Practices report, the author stated: "I consider that a credible threat." And in [P3651R0](https://wg21.link/p3651r0)<sup>[17]</sup> (2025): "Unless we act now and get C++ onto a track supporting a flexible framework of profiles, we risk a painful decline."

WG21 was deliberately designed so that no single individual could dominate. There is no benevolent dictator for life. The committee operates by consensus across study groups, working groups, and plenary votes. This structure serves C++ well for careful, considered language evolution. But it creates a vulnerability: WG21 is a deliberative body that moves at the pace of consensus. It has no executive power. The regulatory threat is not a language-design problem that can be solved over three standard cycles - it is an external political event that moves on a policy timeline. The committee's consensus-driven process is structurally ill-equipped to counter a threat that operates on a political calendar.

This is why profiles carry extraordinary urgency. Profiles are not merely a technical improvement - they are the mechanism by which C++ acquires the capacity for rapid, credible response that WG21's deliberative structure otherwise lacks. Once the profiles framework is approved and implemented, the committee has a standing instrument: new safety guarantees can be defined, named, and enforced without further language changes. The framework is the executive power. Individual profiles are the executive actions. Without the framework, every safety response requires a new multi-year standardization effort. With it, C++ can respond to regulatory demands as they arise.

### 1.3 Confronting Objections

Four objections against standardizing the type-safety profile deserve direct response.

**The concurrency exclusion.** The type-and-resource profile assumes no data races (Section 4). In any multithreaded program, the guarantee carries an asterisk. A partial guarantee is still valuable: the single-threaded subset of a program is where type-safety and lifetime violations concentrate, and eliminating them there reduces the attack surface that data races can exploit. A separate concurrency profile is planned; the type-safety profile does not pretend to solve that problem.

**The false positive rate.** The conservative analysis rule (Section 2.6) rejects valid code when control flow is non-trivial. The false positive rate is unknown but now measurable. [P4137R0](https://wg21.link/p4137r0)<sup>[42]</sup> (PAVE) provides a three-phase methodology: Phase 1 classifies functions mechanically using AST predicates, Phase 2 distinguishes true positives from false positives (annotatable vs. structural), and Phase 3 measures the annotation dividend from `[[not_invalidating]]`. The methodology can be executed today using existing Clang tooling. Design can proceed in parallel with measurement.

**The trusted-code escape hatch.** The implementations of `vector`, `unique_ptr`, `string`, and every standard container require pointer arithmetic, casts, or manual memory management. The profile cannot verify them. They must be marked as trusted and excluded. PAVE Phase 1 measures this exclusion directly. For leaf-application code that uses standard containers without reimplementing them, the trusted boundary is small and auditable.

**Existing analyzers already cover this.** clang-tidy implements five of the profile's eight rules<sup>[42]</sup>. But these checks are non-portable, non-composable, and do not add up to a named guarantee. The profile is the standard; the analyzers are implementations. A guarantee that passes one analyzer but fails another is not a guarantee.

## 2. Specification

As an example, consider

- **Type safety**: Every object is used exclusively according to its definition.
- **Resource safety**: No resource is leaked.

That's the definition of two profiles. In principle, from these simple definitions, we could implement Profiles to enforce these requirements just as we write compilers and libraries to implement specifications. That is, the detailed requirements can in most cases be deduced from the general requirements. However, it is not fair, safe, or manageable to leave the interpretation of these requirements to implementers. In particular, we need a clear definition of "object" and "resource." Beyond that, we need detailed implementation guidance to avoid implementers (possibly ourselves) having to repeatedly reinvent techniques and rediscover problem areas.

The border between what should be specified in the standards document and what should be in an implementation guide is not trivial to define. Different profiles will require different degrees of detail.

That's not a new problem. However, not everybody - not even in WG21 - is used to thinking about this in the context of defining a language feature or a library component.

Code examples can be found in the references.

### 2.1 Object

An object is a region of memory that has been given a type and been initialized according to the definition of that type.

Accessing an object exclusively according to its definition implies that we must not access

- an area of memory as an object before its initialization/construction has completed (except from within its constructor). That is, before it becomes an object.
- an object after its destruction has started (except from within its destructor). That is, after it ceases to be an object.
- an area of memory that has not been initialized/constructed according to the type we use to access it. That is, where it might not be an object, or it might be an object of a different type.

That last point implies that we must eliminate range errors and access through dangling pointers.

### 2.2 Resource

A resource is an entity that must be acquired before use and released after its last use.

That's a very broad definition that cannot be directly translated into a verifiable rule. For example, a `FILE` (a C stream) is a resource because it needs to be opened before use and closed (possibly implicitly at the end of a program). Even an `int` can be a resource. For example, it could be required that it be increased when a scope is entered and decreased as that scope is exited.

To achieve this in a reasonably simple way, we must require that to be considered a resource an entity must have a constructor and a destructor that manages it. That is, an entity that requires explicit acquisition and/or explicit release is simply not considered a resource until it has been encapsulated in a type with a constructor/destructor pair.

### 2.3 Constructors/Destructors

A constructor is supposed to correctly initialize an object, and a destructor is supposed to release every resource. To what extent are we willing to trust that?

We can ensure that every member is initialized in a constructor (except members explicitly marked to remain uninitialized). There should be a separate profile to guarantee initialization for people who cannot adopt the complete type-and-resource profile. That initialization profile should then be used as part of the type-and-resource profile.

The easiest way of ensuring that a destructor matches a constructor is to use only resources managed by constructor/destructor pairs. That is, have only members of types that do not require explicit destruction. For example, using a `vector` or a `unique_ptr` rather than a (pointer, size) pair to represent owned memory. However, that solution doesn't cover every case, e.g., `unique_ptr`'s constructors and destructors.

We can

- Trust the definition of constructors and destructors to be correct (e.g., not leak).
- Require non-trivial constructor/destructor pairs to use explicitly unverified code.

We recommend the former: trust constructors and destructors. The alternative - requiring every non-trivial constructor/destructor pair to be annotated as unverified - would impose a pervasive annotation burden on code that is overwhelmingly correct in practice. The RAII idiom is the most successful resource-management technique in C++; a profile that distrusts it would undermine the very programming style it seeks to enforce. If experience shows that constructor/destructor bugs are a significant source of escapes from the profile's guarantee, this decision should be revisited. PAVE Phase 2<sup>[42]</sup> can measure the rate of resource-safety violations that originate in trusted constructors and destructors.

How do we handle errors in destructors caused by an inability to release a resource? For example, a socket that can't be closed because the other end of a communication channel won't cooperate. We can handle that with the techniques we do today, such as transferring the obligation to close an uncooperative socket to some other code, e.g., by putting it on a to-be-handled-later error list.

### 2.4 Copies and Moves

To be handled similar to constructors and destructors.

### 2.5 Ranges and `nullptr`

The two obvious ways of avoiding accessing an object that is not of the right type (or no type at all) is to prevent

- Range errors
- Access through `nullptr`

This is (relatively) easily done by using hardened libraries doing range checking for types for which the number of elements is known (e.g., `vector` and `span`<sup>[13]</sup>) and a pointer type that is checked for `nullptr` (e.g., `gsl::not_null`<sup>[5]</sup>).

That solves half of the problem. The other half is to ensure that unsafe operations are not used (e.g., subscripting a pointer for which the number of elements pointed to is unknown). For that, we need simple static analysis (e.g., the compiler) to ban or restrict

- Pointer arithmetic
- Subscripting raw pointers

Together, these two approaches are a classical example of the subset-of-superset technique. Without the hardened library, we couldn't ban those basic uses of raw pointers and without the ban of those basic uses of raw pointers we couldn't guarantee the absence of range errors and `nullptr` violations.

### 2.6 Invalidation

There is a nasty third way that we might access an object of the wrong type (or no type at all):

- access through an invalidated pointer; that is, a pointer to an object that had been deleted or gone out of scope

That's also known as the dangling pointer problem. Preventing invalidation by run-time measures implies complications and overheads (e.g., garbage collection). However, it is possible to eliminate access through invalid pointers through static analysis<sup>[14, 15, 5]</sup>. That technique has been prototyped and is currently used as a guideline (at best partly enforced).

Here is an example that must be rejected because it invalidates:

```cpp
void f(vector<int>& v)
{
    v.push_back(7);
}

void g()
{
    vector<int> v = {1, 2};
    auto p = &v[1];
    f(v);
    *p = 3;    // must be prevented
}
```

The problem/error occurs because an alias (`p` for `&v[1]`) is used after the object it referred to has been moved. The basic solution is relatively simple and completely compile time.

When analyzing a function:

- Don't access through an invalid pointer.
- Don't let a pointer to a deleted object or to an object that is about to be destroyed out of the function.
- If an alias to an object has been taken, don't call a function that can invalidate it.
- Return only pointers that are known to:
  - Have been received as arguments
  - Point to something static
  - Point to something on the free store

By "pointer", we include anything that refers to an object (e.g., references and lambda captures). By "return", we include any way of passing a pointer out of a scope (e.g., assigning to a static object).

When analyzing a set of functions:

- Assume that a non-`const` function and a function accessing an object through a non-`const` pointer argument invalidates.
- Assume that a `const` function and a function accessing an object through a `const` pointer argument doesn't invalidate.
- Add a `[[not_invalidating]]` attribute to be used to speed up analysis by marking non-`const` functions and functions that take non-`const` pointer arguments that don't invalidate (e.g., `vector<T>::size()`)<sup>[16]</sup>.

Note that a `[[not_invalidating]]` attribute can be verified when the function definition is compiled. It is not an annotation that relies on programmers using it correctly, just an optimization to save work for the compiler.

The implementation problem here is that what is returned from a function and what has been deleted can be flow-dependent. So, we must accept only code that's not too complex to analyze.

- Some examples are fundamentally impossible to statically analyze.
- Some examples are so complex that a static analyzer would slow down compilation to an unacceptable degree.

An example of code that is fundamentally too complex to accept:

```cpp
int glob = 7;

int* f(int i)
{
    int x = 5;
    return (0 < i) ? &glob : &x;  // might return pointer
                                   // to destroyed object
}
```

We need implementer input to choose what limitation is practical. In particular, to preserve portability, we need a definition of "too complex" so that all compilers agree on what to reject. I suggest:

- If a condition cannot be evaluated at compile time, whatever is guarded by that condition is assumed to be executed.

That is, conditional use of some constructs is treated as unconditional:

- creation of an alias,
- the return of an invalid pointer,
- the invalidation of a pointer,
- the call of an invalidating function, or
- the use of an invalid pointer.

The alternative - flow-sensitive analysis that tracks each branch independently - would provide tighter results but at the cost of portability and compilation time. Different compilers would reach different conclusions about the same code, undermining the guarantee's portability. The conservative rule is sound: it produces no false negatives. If PAVE Phase 2<sup>[42]</sup> measurements show the structural false positive rate is unacceptable, the rule should be relaxed in a future revision at the cost of more complex analysis.

In older code and code written in older styles, this limitation will be a burden, but encouraging the use of simpler code is beneficial in its own right.

## 3. The Profiles Framework

The Profiles framework (based on the subset-of-superset strategy described in Section 1<sup>[8, 9]</sup>) has been developed over several years to cover a wide range of requests involving a variety of notions of safety and style involving combinations of compile-time and run-time techniques<sup>[17]</sup>. The Framework, complete with syntax and working paper wording, is described in detail in [P3589R2](https://wg21.link/p3589r2)<sup>[6]</sup>. In particular, it introduces notation for enforcing, requiring, and suppressing profiles.

The framework is to be common to all profiles and independent of any particular profile.

## 4. Other Profiles

To be complete the type-and-resource profile will depend on other profiles. To allow gradual adoption of profiles, such profiles should be independently defined so that they can be used on their own<sup>[18]</sup>. Examples of possible standard profiles are:

- **initialization**: every object is properly constructed/initialized (or explicitly declared not to be).
- **casting**: no old-style non-verifiable casts.
- **union**: no union use.
- **ranges**: all pointer arithmetic and all subscripting are checked for range errors.

Obviously, when some construct is banned (outside unverified/trusted code), alternatives must be provided, e.g., `variant`s instead of `union`s. The profiles listed above are relatively easy to specify and thus good candidates for early implementation.

Most likely separate from the type-and-resource profile, but related, we need:

- **arithmetic**: eliminate implicit narrowing conversions, overflow, and underflow.
- **algorithms**: use only ranges (rather than iterator pairs) and prevent dereferencing of `end()` iterators.
- **concurrency**: The type-and-resource profile assumes that there are no data races. To ensure that, a separate concurrency profile is needed. Specifying that is non-trivial and beyond the scope of this paper. There must eventually be a concurrency profile and part of that will undoubtedly be ways of minimizing sharing<sup>[5]</sup>.

These profiles are important, but harder to specify. Note that libraries for checking arithmetic exist (e.g.<sup>[19, 20]</sup>).

## 5. Developer Experience

Three questions determine whether developers will adopt profiles.

**How does a developer encounter profiles for the first time?** A profile is activated per [P3589R2](https://wg21.link/p3589r2)<sup>[6]</sup> using `[[enforce(type_safety)]]` on a declaration or translation unit, or via a compiler flag (e.g., `-fprofile=type_safety`). The activation is visible in the source, making intent discoverable by code review and tooling.

**What happens when the profile rejects valid code?** Consider the conditional-return example from Section 2.6. The conservative rule rejects it because both branches are assumed to execute. A plausible diagnostic:

```
error: function 'f' may return a pointer to local
    variable 'x' [type_safety.invalidation]
    return (0 < i) ? &glob : &x;
                              ^
note: under the type_safety profile, conditional
    returns of local pointers are treated as
    unconditional
```

The developer's recovery path: refactor into two functions (one returning `&glob`, one with `x` scoped appropriately), or if the function genuinely does not invalidate, add `[[not_invalidating]]` and let the compiler verify the claim.

**Which mental models do profiles reinforce, and which do they break?** Profiles reinforce RAII and `const` correctness - developers who already follow these idioms will find most of their code passes unchanged. The model that breaks is the assumption that "my non-`const` function is fine because I know it does not invalidate." The profile assumes non-`const` means invalidating unless annotated otherwise. For developers accustomed to writing correct-but-unannotated non-`const` functions, the transition cost is adding `[[not_invalidating]]` where the compiler can verify it. This is a one-time cost that produces a machine-checkable claim.

---

## 6. Domain Applicability

The type-safety profile affects multiple domains:

- **Leaf applications** (web services, desktop apps, CLI tools) using standard containers and smart pointers. This is the profile's primary target. Most code in these domains avoids pointer arithmetic and C-style casts.
- **Finance and enterprise** code values correctness guarantees and has resources for adoption. Legacy codebases may require incremental migration.
- **Games** use performance-sensitive code with raw pointers and unions. The trusted-code boundary will be larger.
- **Embedded systems** rely heavily on pointer arithmetic, memory-mapped I/O, and C interop. Much of this code will fall in the trusted category.
- **Scientific computing** uses pointer arithmetic on arrays pervasively. Similar to embedded.
- **Systems programming** (OS kernels, drivers, allocators) is where the abstraction implementations live. This code is excluded by design.

The profile was designed for leaf-application code. Domains that rely on banned constructs will have a larger trusted-code boundary. This is intentional: the profile does not claim to verify the implementation of `vector`, only the code that uses it.

PAVE<sup>[42]</sup> can be run on codebases from each domain. The Phase 1 distribution (clean/rejected/trusted) will differ - a web-service application will show higher coverage than an embedded systems library. Both measurements are informative, and the difference quantifies the domain coverage gap.

## 7. Implementation Evidence

What is described has been prototyped<sup>[4, 5, 6]</sup> and an experimental implementation is being conducted in a major C++ compiler. An implementation guide is being written.

The profile's rules map to well-understood AST patterns. [P4137R0](https://wg21.link/p4137r0)<sup>[42]</sup> documents that seven of the eight rules are syntactic pattern matches against the Clang AST. Five already have clang-tidy implementations:

| clang-tidy Check                                      | Profile Rule             |
| ----------------------------------------------------- | ------------------------ |
| `cppcoreguidelines-pro-bounds-pointer-arithmetic`     | No pointer arithmetic    |
| `cppcoreguidelines-pro-bounds-array-to-pointer-decay` | No raw pointer subscript |
| `cppcoreguidelines-pro-type-reinterpret-cast`         | No `reinterpret_cast`    |
| `cppcoreguidelines-pro-type-cstyle-cast`              | No C-style cast          |
| `cppcoreguidelines-pro-type-union-access`             | No union access          |

The code examples in Section 2.6 are not hypothetical - they are canonical instances of PAVE's classification pipeline. The conditional-return example (`return (0 < i) ? &glob : &x;`) maps to the "no local pointer return" AST predicate. The `push_back` invalidation example maps to the "no invalidating call with alias" predicate. Both can be classified mechanically by a PAVE Phase 1 tool without evaluating conditions, demonstrating that the rules are implementable with existing AST infrastructure.

Interactions with modules, coroutines, and `constexpr` remain to be tested in the experimental implementation. These are open edges that need implementer input.

## 8. Success Criteria

The following metrics define what successful adoption of the type-safety profile looks like. PAVE<sup>[42]</sup> provides the measurement methodology for the first three.

- **Coverage rate**: The profile classifies at least N% of functions as "clean" in a representative leaf-application codebase (PAVE Phase 1).
- **False positive rate**: The structural false positive rate (PAVE Phase 2) is below M%. The conditional-return example in Section 2.6 is the canonical instance - PAVE Phase 2 determines whether such patterns are genuinely unsafe or whether the conservative analysis is too aggressive.
- **Annotation dividend**: The gap between Phase 1 coverage and Phase 3 coverage (after `[[not_invalidating]]` annotations are inferred) is measurable. If large, it validates the annotation design. If small, the rules themselves need revision.
- **Implementation convergence**: At least two major implementations ship profile support within two standard releases of adoption.
- **CVE impact**: Type-safety-related CVE frequency declines measurably in codebases that adopt the profile.

## 9. Testable Predictions

The design rests on predictions that should be recorded with falsifiable criteria.

| Prediction                                              | Falsifiable Test                                          | Revisit                    |
| ------------------------------------------------------- | --------------------------------------------------------- | -------------------------- |
| Safety without inessential overhead                     | Benchmark profiled vs. unprofiled code                    | Before IS adoption         |
| Conservative analysis is practical                      | PAVE Phase 1: clean rate exceeds N% on leaf-app codebase | Before IS adoption         |
| Trusted-code boundary is small for applications         | PAVE Phase 1: trusted rate below M% on leaf-app codebase | Before IS adoption         |
| `[[not_invalidating]]` reduces false positives          | PAVE Phase 3 minus Phase 1 (annotation dividend)         | Two releases post-adoption |
| Structural false positive rate is manageable            | PAVE Phase 2: structural FP rate below K%                | Before IS adoption         |
| Implementations converge without incompatibilities      | Cross-compiler comparison on a shared test suite          | Two releases post-adoption |

PAVE<sup>[42]</sup> provides the methodology for four of the six predictions. These can be measured today using existing Clang tooling.

### Retrospective Commitment

The committee should revisit the type-safety profile's design decisions two standard releases or six years after adoption, whichever comes first. The questions:

- Was the conservative analysis practical or too aggressive? (PAVE Phase 1 clean rate on real codebases.)
- Did `[[not_invalidating]]` suffice to reduce false positives? (PAVE annotation dividend.)
- Did implementations converge on the same acceptance/rejection decisions? (Cross-compiler test suite.)
- Is the concurrency exclusion still acceptable, or must a concurrency profile be required?

The PAVE methodology provides the concrete measurement tools for three of the four questions. The retrospective is not aspirational - it has a defined execution plan.

---

## 10. What Needs to Be Done Soon

To make progress, we need to specify, implement, and try out a few simple profiles to gain experience. To avoid chaos, we urgently need the framework to be approved<sup>[21]</sup>. Many ideas that are part of profiles have been prototyped and/or used, but to make progress we must build on the proposed Profiles framework [P3589R2](https://wg21.link/p3589r2)<sup>[6]</sup> and start experimenting with specific profiles within that framework.

For that to happen, we need the framework available and accessible on one or more C++ implementations, ideally on all major implementations. Otherwise, using different initial/experimental profiles will require too much boilerplate code and different interfaces to different tool chains (e.g., in-code annotations, compiler options, and build-system settings). That will impede portability and possibly introduce incompatibilities that will be hard to root out in the future. Currently, there is - to the best of my knowledge - a specification<sup>[6]</sup> and an experimental implementation is being conducted in a major C++ compiler. Also, an implementation guide is being written. We encourage implementers to coordinate and if possible, collaborate to avoid inessential incompatibilities<sup>[21]</sup>.

Beyond the framework implementation(s), we need to design and precisely specify a few initial profiles. We need a reasonable common style of Profile descriptions. That - and bringing the older documents in line with the C++26 specification - is a necessary first step on the way to standardization. A start is made in [P3274R0](https://wg21.link/p3274r0)<sup>[7]</sup>.
